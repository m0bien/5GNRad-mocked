# 5G NR PRS Radar (5GNRad) — Documentation

## Purpose

This repository provides an end-to-end **link-level simulation workflow** to evaluate **sensing with cellular infrastructure** using:

- **Standard-compliant 5G NR Positioning Reference Signals (PRS)** as the sensing waveform
- **3GPP multipath channel descriptions** (environment/background + optional target) as the propagation model
- A radar processing chain that produces **range–Doppler–angle detections** and estimates target location/velocity

The emphasis is **protocol-compliant sensing**: the transmitted signal is generated from NR PRS resource definitions, so results reflect real deployment constraints (comb patterns, symbol mapping, periodicity, etc.).

---

## Quick start

From the repository root:

```matlab
main
```

To run a single scenario:

```matlab
scenarioPath = "examples/uma_trp1_3gpp";

[simConfig, stConfig, prsConfig, geometry, sensConfig, backgroundChannel, targetChannel] = ...
    nrRadar.cfg.configureScenario(scenarioPath);

[results, detStats, detectionOutput, info] = nrRadar.run( ...
    simConfig, stConfig, prsConfig, geometry, sensConfig, ...
    backgroundChannel, targetChannel, [], ...
    "Parallel", "off");

nrRadar.io.exportResults(scenarioPath, results, detStats, detectionOutput);
```

## Scenario input files

Each scenario under `examples/<scenarioName>/Input/` is described by a small set of **text** and **JSON** files.

### Common format for `*.txt` configuration files

- Tab-delimited `Parameter<TAB>Value` pairs
- Values are parsed as strings first, then validated/cast to numeric/enums by the configuration layer

Example:

```
dopplerFftLen\t256
window\ttaylor
```

### `simulationConfig.txt`

Defines system-level parameters and runtime controls.

**Parameters** (MathWorks style):

- **systemFc** — Carrier frequency (Hz). Default: `6e9`.
- **systemBw** — RF bandwidth (Hz). Allowed: `[10 20 40 50 60 80 100 200 400]e6`. Default: `100e6`.
- **systemNF** — Receiver noise figure (dB). Allowed: `[0, 20]`. Default: `7`.
- **systemN0** — Noise PSD override (dBm/Hz). If non-empty, overrides noise computed from `systemNF`. Default: `[]`.
- **txPower** — Transmit power (dBm). Allowed: `[-20, 80]`. Default: `23`.
- **carrierSubcarrierSpacing** — NR SCS (kHz). Allowed: `{15, 30, 60, 120, 240}`. Default: `30`.
- **carrierNSizeGrid** — Carrier grid size (RBs). Allowed: `[1, 273]`. Default: `273`.
- **carrierNStartGrid** — Carrier start RB. Allowed: `[0, 2199]`. Default: `0`.
- **channelScenario** — Scenario label (string). Allowed: `{UMiAV, UMaAV, RMaAV}`. Default: `UMiAV`.
- **antennaCouplingEfficiency** — Linear coupling efficiency `[0, 1]`. Default: `1`.
- **nStDrop** — Number of target states processed per drop (integer). Default: `1`.
- **nMaxDrop** — Max number of drops to run (integer). If larger than available channel/target states, it is clipped. Default: `50`.
- **trpYawDeg** — TRP yaw rotation about global z-axis (deg). Allowed: `[-180, 180]`. Default: `0`.
- **maxRangeInterest** — Maximum range of interest (m). Used to size/crop processing bins. Default: `200`.

**Notes**

- `txAntennaModel.json` and `rxAntennaModel.json` are required files and are loaded alongside `simulationConfig.txt`.

### `prsConfig.txt`

Defines the NR PRS resource used as sensing waveform.

**Parameters**:

- **PRSResourceSetPeriod** — PRS resource periodicity `[period slotOffset]` (slots). Default: `[1 0]`.
- **PRSResourceOffset** — PRS resource offset (integer). Default: `0`.
- **PRSResourceRepetition** — Repetition factor (integer). Default: `1`.
- **PRSResourceTimeGap** — Time gap between PRS resources (integer). Default: `1`.
- **NumRB** — Number of PRS RBs. Default: `51`.
- **RBOffset** — PRS RB offset. Default: `0`.
- **NumPRSSymbols** — PRS symbols per slot. Allowed: `{2,4,6,12}`. Default: `12`.
- **SymbolStart** — First PRS symbol index in a slot. Allowed: `[0, 13]`. Default: `0`.
- **CombSize** — PRS comb size (subcarrier spacing pattern). Default: `2`.
- **PRSResourceBlock** — PRS resource block index (integer). Default: `0`.

### `sensConfig.txt`

Defines processing parameters (FFT sizes, CFAR settings, DOA method, etc.).

**Core processing**

- **numberSensingSymbols** — Number of PRS-bearing symbols used in slow-time processing. Default: `10`.
- **dopplerFftLen** — Doppler FFT length. Default: `256`.
- **window** — Doppler window type. Allowed: `{rect, hamming, blackmanharris, hann, taylor, gaussian}`. Default: `taylor`.
- **azFftLen** — Azimuth FFT size for beamspace processing. Default: `256`.
- **elFftLen** — Elevation FFT size for beamspace processing. Default: `256`.
- **doaEstimationMethod** — DOA method. Allowed: `{bartlett, beamspacefft}`. Default: `beamspacefft`.

**CFAR (2-D range–Doppler)**

- **cfarGrdCellRange** — Guard cells in range. Default: `4`.
- **cfarTrnCellRange** — Training cells in range. Default: `4`.
- **cfarGrdCellVelocity** — Guard cells in Doppler. Default: `6`.
- **cfarTrnCellVelocity** — Training cells in Doppler. Default: `6`.
- **cfarThreshold** — Threshold scale (linear multiplier on noise estimate). Default: `10`.

**Experimental**

These parameters exist in the configuration layer but are not used by the current default detection path (they are kept for forward-compatibility and experimentation):

- **rdaThreshold** — Optional pre-thresholding of the RD map before CFAR.
- **nmsMaxPeaks**, **nmsRadius** — Optional alternative NMS controls.
- **cfarGrdCellAzimuth**, **cfarTrnCellAzimuth**, **cfarGrdCellElevation**, **cfarTrnCellElevation** — Angle-domain CFAR parameters.
- **dbscanEps**, **dbscanMinPts**, **dbscanNorm** — Exposed in config, but default DBSCAN settings are currently applied inside `nrRadar.sens.rdmDetection`.
- **dbscanMinMaxRatio** — Used to suppress weak clusters relative to the strongest cluster.

### `bsConfig.txt`

Base station / TRP configuration.

- A single row with **Cartesian coordinates** `[x, y, z]` in meters (scenario frame).

### `targetConfig.txt`

Target state truth for the scenario.

- CSV file, each row: `[x, y, z, vx, vy, vz]`
- Units: meters and meters/second

The simulator groups rows by `nStDrop` to create each “drop”. For example, with `nStDrop = 5`, each drop uses 5 consecutive target states.

## Antenna model JSON

Antenna and beamforming are described by `txAntennaModel.json` and `rxAntennaModel.json`.

### Schema (high level)

```json
{
  "meta": {
    "name": "TxUMa_Digital",
    "fcHz": 6000000000,
    "type": "phased.URA",
    "array": { "M": 8, "N": 8, "dV_lambda": 0.5, "dH_lambda": 0.5 }
  },
  "beamformer": {
    "type": "digital",
    "wElem": [[...], [...]]
  }
}
```

### Fields

**meta**

- **meta.name** — Human-readable identifier.
- **meta.fcHz** — Design frequency used for metadata (Hz).
- **meta.type** — Array type (e.g., `phased.URA`).
- **meta.array.M, meta.array.N** — URA dimensions.
- **meta.array.dV_lambda, meta.array.dH_lambda** — Element spacing in wavelengths.

**beamformer**

- **beamformer.type** — One of `{digital, hybrid, analog}`.
- **beamformer.wElem** — Element weights.
 
> Implementation note: The JSON is imported by `nrRadar.io.importBFfromJSON`. Complex matrices are stored as `[Re, Im]` pairs and reconstructed inside MATLAB.

## Channel file specification (JSON)

The simulator can load externally generated multipath descriptions.

- `backgroundChannel.json` — environment/clutter MPCs
- `targetChannel.json` — target MPCs tied to a moving object

### Common conventions

- Delays are in **seconds**.
- Angles use the **3GPP CDL convention**:
  - `AnglesAoA`, `AnglesAoD` in **degrees** (azimuth).
  - `AnglesZoA`, `AnglesZoD` in **degrees** (zenith angles).
  - Internally, elevation is computed as `el = 90 - Zo`.
- `AveragePathGains` are expected in **linear power scale** (not dB). The simulator converts them to dB internally.
- `InitialPhases` are in **radians**.

### `backgroundChannel.json`

- JSON array of length `NDrop`.
- Each entry describes an environment multipath set for a given drop.

Each element contains:

- **PathDelays** — `[P×1]` (seconds)
- **AveragePathGains** — `[P×Npol]` (linear power)
- **AnglesAoD**, **AnglesAoA** — `[P×1]` (deg)
- **AnglesZoD**, **AnglesZoA** — `[P×1]` (deg)
- **InitialPhases** — `[P×Npol]` or `[P×1]` (radians)
- **HasLOSCluster** — scalar logical/0-1
- **CPM** — optional (may be empty)

### `targetChannel.json`

- JSON array aligned with the target truth timeline.
- Each element describes the **target multipath** at a given state/time index.

Each element contains the same fields as background channel (above).

> Implementation note: Channel construction is performed by `nrRadar.channel.getSensingCdl`, which merges background and target MPCs, applies TRP yaw rotation, and returns an element-domain CIR tensor.

# System model

This section summarizes the signal and observation model implemented by the simulator.

## A. Architecture

We consider a **monostatic sensing** configuration where a base station (TRP) transmits NR PRS and receives reflections from:

- the environment/clutter (`backgroundChannel`), and
- a moving target(s) (`targetChannel`)

Transmit and receive arrays are co-located at the TRP and can support analog, hybrid, or digital beamforming.

## B. PRS–OFDM signal model

PRS is defined on an NR OFDM grid. Only a subset of subcarriers is active, determined by the PRS comb pattern.

Let `k` index subcarriers and `\ell` index OFDM symbols. A simplified model for PRS mapping is:

$$
 s_{k,\ell} =
 \begin{cases}
 \beta_{\mathrm{PRS}}\,c_{k,\ell}, & k \bmod K = 0 \\
 0, & \text{otherwise}
 \end{cases}
$$

where:

- $c_{k,\ell}$ is the QPSK PRS sequence element
- $K$ is the PRS comb size
- $\beta_{\mathrm{PRS}}$ is a normalization factor

The time-domain waveform is generated by OFDM modulation (IFFT + cyclic prefix insertion) using NR numerology parameters.

## C. Beamforming model

Let $\mathbf{w}_{\mathrm{tx}}$ and $\mathbf{W}_{\mathrm{rx}}$ denote transmit and receive beamformers.

The transmitted signal can be written as

$$
\mathbf{x}(t) = \mathbf{w}_{\mathrm{tx}}\,s(t).
$$

The received digital-chain signals are

$$
\mathbf{y}(t) = \mathbf{W}_{\mathrm{rx}}^H \sum_{p=1}^{P} \alpha_p\, \mathbf{a}(\theta_p,\phi_p)\, s(t-\tau_p)\,e^{j2\pi f_{D,p} t} + \mathbf{n}(t)
$$

where $\alpha_p,\tau_p,f_{D,p}$ are gain/delay/Doppler for MPC $p$, and $\mathbf{a}(\theta_p,\phi_p)$ is the array response.

## D. Channel model loading

The simulator loads multipath descriptions from JSON rather than generating them internally. This allows:

- replaying deterministic, externally generated 3GPP channels
- swapping in measured channel extractions

## E. Radar processing products

For each drop, the simulator forms:

1. **Channel estimate** on PRS resources
2. **Range processing** via IFFT across frequency to obtain delay/range profiles
3. **Doppler processing** via slow-time FFT across repeated PRS symbols
4. **Angle processing** per detection using either beamspace FFT or Bartlett scan

## F. Geometry reconstruction

After estimating range $\hat r$, azimuth $\hat\theta$, and elevation $\hat\phi$, the simulator computes a 3D scattering-point estimate by intersecting the measurement ray with the scenario geometry.

# Code architecture and main building blocks

## Top-level workflow

The recommended entry point is `main.m`:

1. `setup()` adds source folders to the MATLAB path
2. Iterate scenario folders under `examples/`
3. `nrRadar.cfg.configureScenario(scenarioPath)` loads and validates configuration
4. `nrRadar.run(...)` executes the simulation
5. `nrRadar.io.exportResults(...)` writes output products under `Output/`

## Main execution structure

### `nrRadar.run`

```matlab
[results, detStats, detectionOutput, info] = nrRadar.run( ...
    simConfig, stConfig, prsConfig, geometry, sensConfig, ...
    backgroundChannel, targetChannel, desiredWorkers, ...
    "Parallel", "auto", "PoolProfile", "Processes", "ReusePool", true);
```

**Name-value arguments**:

- **"Parallel"** — `'auto' | 'on' | 'off'` (default: `'auto'`).
- **"PoolProfile"** — Parallel pool profile name (default: `'Processes'`).
- **"ReusePool"** — Reuse an existing pool when possible (default: `true`).

`nrRadar.run` returns:

- `results` — struct array (one per drop)
- `detStats` — summary table/struct of key metrics
- `detectionOutput` — detailed per-drop payloads (positions, velocities, peak metrics)

## Data flow

Conceptually, the simulator is split into:

1. **Precompute** (once per scenario)
   - PRS grid + waveform
   - OFDM constants (FFT size, CP lengths, sample rate)
   - range and velocity bin vectors
   - beamspace grids for angle processing

2. **Per-drop processing** (serial or `parfor`)
   - build sensing CIR (background + optional target)
   - convolve TX waveform with the channel
   - estimate frequency-domain channel on PRS resources
   - form range–Doppler map
   - detect peaks and estimate angles
   - reconstruct 3D positions and score against truth

Key functions:

- `nrRadar.internal.precompute` — builds cached constants/grids
- `nrRadar.internal.dropWorker` — processes a single drop (parfor-safe)
- `nrRadar.channel.getSensingCdl` — merges MPCs and returns CIR
- `nrRadar.rx.getRxWaveform` — waveform convolution + noise
- `nrRadar.rx.estimateChannel` — OFDM demod + PRS-based channel estimate
- `nrRadar.sens.getRangeDoppler` — RD formation + slow-time mean removal
- `nrRadar.sens.rdmDetection` — CFAR, peak picking, DOA, clustering, reconstruction

# Detection and estimation algorithms

This section describes the default sensing algorithm implemented by `nrRadar.sens.rdmDetection`.

## 1) Range–Doppler map formation

Given a complex range–Doppler tensor $X[r,d,n]$ (range bin, Doppler bin, digital chain index), the detector forms a 2-D power map by projecting across digital chains:

$$
P[r,d] = \sum_n |X[r,d,n]|^2.
$$

A slow-time **mean removal** is applied during Doppler processing to reduce stationary clutter.

## 2) 2-D CA-CFAR

A 2-D CA-CFAR is applied on $P[r,d]$ with guard and training regions in range/Doppler.

- Guard: `cfarGrdCellRange`, `cfarGrdCellVelocity`
- Training: `cfarTrnCellRange`, `cfarTrnCellVelocity`
- Threshold scale: `cfarThreshold`

The detector returns a binary detection mask and an estimated local noise level for each CUT.

## 3) Peak picking (NMS)

CFAR detections can be contiguous blobs. The implementation performs non-maximum suppression (NMS) using a suppression radius derived from the CFAR guard region.

Output: a sparse list of RD peaks $(r_i, d_i, v_i)$.

## 4) Angle estimation per peak

For each detected $(r_i,d_i)$:

- Extract the complex vector across digital chains
- Estimate DOA using one of:

**A) Beamspace 2-D FFT (`doaEstimationMethod = beamspacefft`)**

- Reshape the detection vector into an $M\times N$ URA grid
- Apply a 2-D FFT to form beamspace response
- Select the maximum beam

**B) Bartlett scan (`doaEstimationMethod = bartlett`)**

- Build steering vectors on an az/el grid
- Compute matched-filter power for each direction
- Select the argmax


## 5) 4-D clustering and sidelobe suppression

Peaks are represented in 4-D bin space:

$$
\mathbf{p}_i = [r_i, d_i, e_i, a_i].
$$

A DBSCAN clustering groups nearby peaks into object-level detections. 

## 6) Parameter estimation and geometry reconstruction

For each final detection:

1. Convert bin indices to physical estimates (range, velocity, azimuth, elevation)
2. Compute a 3D Cartesian scattering-point estimate
3. Reject invalid solutions (NaNs, geometry constraints)

Outputs per drop include estimated positions, radial velocities, and a post-processed SNR estimate.

# Outputs

## In-memory outputs (`nrRadar.run`)

`results` is a struct array with one element per drop. Each element includes:

- Drop metadata: `q`, `targetIdx`, `timeIndex`
- Error metrics: `positionErrorX/Y/Z`, `positionErrorH/V`, `rangeError`, `velocityError`, `azimuthError`, `elevationError`
- Detection counts: `TP`, `FN`, `FP`, `FPR`, `nDetectedTarget`
- Signal quality: `snrSlice`
- Detailed payload: `detectionPayload` (ground truth + detections)

## Exported outputs (`nrRadar.io.exportResults`)

`exportResults` writes `Output/` under the scenario folder. Existing `Output/` is deleted and recreated.

### `Output/detStats.csv`

Per-drop summary table. Typical columns:

- `timeIndex` — drop index
- `snrSlice` — SNR (dB)
- `FPR`, `FN`, `FP`, `TP` — detection metrics

### `Output/error.csv`

Per-target (per state) error table. Typical columns:

- `q` — drop index
- `targetIdx` — target/state index
- `positionErrorX/Y/Z` — Cartesian error components (m)
- `positionErrorH/V` — horizontal/vertical error (m)
- `rangeError` — range error (m)
- `velocityError` — radial velocity error (m/s)
- `azimuthError`, `elevationError` — angle errors (deg)

### `Output/detection.json`

Detailed detection payload per drop. Each JSON element contains:

- `q` — drop index
- `targetIdx` — indices of target states used in this drop
- `gtPosition` — ground truth positions (m)
- `stPositionEstimate` — estimated positions (m)
- `gtVel` — ground truth radial velocity component (m/s)
- `velParallel` — estimated radial velocity (m/s)
- `peakRda` — detection strength metric (implementation-defined)

# Tutorial: running and modifying examples

## 1) Run the full batch

```matlab
main
```

`main.m` iterates through a list of scenarios under `examples/` and runs the full chain.

## 2) Modify a scenario

### Change PRS configuration

Edit `Input/prsConfig.txt`:

- `CombSize`
- `NumPRSSymbols`
- `PRSResourceSetPeriod`

### Change detection sensitivity

Edit `Input/sensConfig.txt`:

- `cfarThreshold`
- guard/training sizes

### Switch DOA method

Set `doaEstimationMethod` to:

- `beamspacefft` (fast)
- `bartlett` (robust)

### Enable parallel execution

In your runner:

```matlab
[results, detStats, detectionOutput, info] = nrRadar.run(..., "Parallel", "on");
```

Optionally set `desiredWorkers` (or leave empty for auto selection).

## 3) Add a new scenario

1. Copy an existing scenario folder.
2. Update `Input/` files:
   - `simulationConfig.txt`, `prsConfig.txt`, `sensConfig.txt`
   - `bsConfig.txt`, `targetConfig.txt`
   - antenna JSON models
   - channel JSONs (optional)
3. Add the folder to the `scenarios` list in `main.m`.

## Performance tips

Runtime is dominated by:

- number of drops
- Doppler FFT length and number of sensing symbols
- angle FFT sizes
- channel length (number of MPCs)

Start with small FFT sizes and fewer drops; scale up once the workflow is validated.
